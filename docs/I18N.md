# I18N.md — Wraithgrove Internationalization Guide

Wraithgrove uses a lightweight i18n system: a single JSON locale file per language, loaded at boot and cached in `localStorage`. The module is `js/core/wg-i18n.js`; the English baseline is `locales/en.json`.

---

## How it works

```
locales/
  en.json         ← English baseline (ship-blocking)
  es.json         ← Spanish (translator-contributed)
  ja.json         ← Japanese (translator-contributed)
```

At boot the game reads `wg_settings_v1.language` from `localStorage` (defaults to `'en'`), calls `WG.i18n.init(code)`, which:
1. Checks `localStorage.wg_locale_<code>` for a cached copy.
2. If absent, fetches `locales/<code>.json` and stores it.
3. Falls back to `en` on any fetch error.

After init, `WG.i18n.refreshDOM()` replaces text in elements carrying `data-i18n="key"` attributes.

---

## API

```js
WG.i18n.t(key)                    // → localized string, falls back to key itself
WG.i18n.t(key, { n: 5 })          // → string with {n} replaced: "5 rewards ready"
WG.i18n.setLocale('es')           // switch locale, refresh DOM, persist choice
WG.i18n.getLocale()               // → 'en' | 'es' | ...
WG.i18n.refreshDOM()              // re-apply all data-i18n attributes in DOM
```

## Adding a new locale (translator guide)

1. **Copy the English baseline:**
   ```bash
   cp locales/en.json locales/es.json
   ```

2. **Translate all values** in the new file. Keys are stable identifiers — never translate the keys themselves.

3. **Template params** — some strings contain `{param}` placeholders. Keep them as-is; only translate the surrounding text:
   ```json
   "results.revive_timer": "La etapa termina en {sec}s"
   ```

4. **Add the option** to the language picker in `js/wg-game.js` by removing the `disabled` attribute from the relevant `<option>`:
   ```js
   '<option value="es"'+(cfg.language==='es'?' selected':'')+'>Español</option>'
   ```
   Also add the locale's display name as a key: `"lang.es": "Español"`.

5. **Test** by opening Settings → Language, switching to the new locale, and verifying top-level UI (nav tabs, BATTLE button, mode pills, boot screen).

6. **Submit a PR** with only `locales/<code>.json` changed plus the option-enable in `wg-game.js`.

## Wrapping new strings (developer guide)

**In JS files:**
```js
// Single string
someEl.textContent = WG.i18n ? WG.i18n.t('my.key') : 'Fallback';

// Inside a render function with a local alias (preferred for many uses)
const _t = WG.i18n ? WG.i18n.t.bind(WG.i18n) : function(k) { return k; };
btn.textContent = _t('my.key');

// With params
msg.textContent = WG.i18n.t('results.revive_timer', { sec: 10 });
```

**In HTML (static elements only):**
```html
<span data-i18n="nav.hunt">Hunt</span>
```
`refreshDOM()` is called after locale loads and after `setLocale()`. Leave the fallback text in the tag body so the page is readable even before JS runs.

**Scope discipline:** Only wrap user-visible text. Debug logs, internal error messages, console output, and data model strings (stage ids, biome names, enemy type ids) must stay literal — they appear in no locale file.

## Cache invalidation

Locales are cached in `localStorage.wg_locale_<code>` and served with a `?v=<BUILD.version>` bust. To force a refresh without a version bump, delete the localStorage key:
```js
localStorage.removeItem('wg_locale_es');
```
