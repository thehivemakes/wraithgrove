## Issue Ranking

### 🔴 CRITICAL — Blocks core gameplay

---

#### C-01 · FORGE tab crash fires on every tab switch

**File:** `js/forge/forge-render.js:655`  
**Error:** `TypeError: Cannot read properties of undefined (reading 'name')`  
**Stack:**
```
makeBuildingTile  forge-render.js:655
makeBuildingsSection  forge-render.js:550
refresh  forge-render.js:529
tab:change subscriber  wg-engine.js:6
setActiveTab  wg-state.js:185
switchTab  wg-game.js:367
```

**Root cause:** `ForgeBuildings.DEFS` was renamed after save data was written. The `wg_save_v2` record contains 6 building IDs that no longer exist in the catalog.

| Save ID (stale) | Current DEFS key | Slot |
|---|---|---|
| `cave` | `gold_mine` | 0 |
| `fence` | `anvil` | 3 |
| `cannon` | `cannon_battery` | 4 |
| `blade` | `bow_range` | 5 |
| `bow` | `barracks` | 6 |
| `sawtrap` | `wall_workshop` | 7 |

Two slots survive unchanged: `forge` (slot 1), `campfire` (slot 2).

**Impact:** forge-render subscribes to the global `tab:change` event, so this TypeError fires on **every navigation action** across all tabs, not just when visiting FORGE. Error is swallowed by the engine's try/catch but FORGE tab never renders its building grid.

**Fix required:** Save migration that rewrites the 6 stale IDs on `wg_save_v2` load, before ForgeBuildings.render() is called.

---

#### C-02 · Game loop stalls within 0.73 s of entering combat

**Symptom:** After `hunt_start` event fires, `WG.Game.getHuntRuntime().elapsed` advances to ~0.73 s then freezes. `delta` reads `0.0000` on successive polls 200 ms apart. No creature spawns, no player animation — screen shows static stage art.

**Confirmed via:**
```js
const t0 = WG.Game.getHuntRuntime();
// 200 ms busy-wait
const t1 = WG.Game.getHuntRuntime();
// t0.elapsed === t1.elapsed === 0.7284
// t1.delta === 0.0000
```

**Scope:** Affects both BATTLE (Stage 1 tested) and GAUNTLET (Floor 1 tested). No console error emitted — the rAF loop silently drops.

**Impact:** All combat gameplay is non-functional. Players reach the combat screen but the game never progresses.

**Fix required:** Identify what causes the rAF callback to stop re-scheduling itself within the first second. Candidates: uncaught exception inside the rAF body silently swallowed, a guard condition that permanently fails, or a flag set by the FORGE crash (C-01) that poisons game state before combat begins.

#### H-01 · Version string mismatch in SETTINGS → ABOUT

**Displayed:** "Unlimited Chaos v0.35.1 · Build 2026-05-08"  
**Deployed bundle version:** `0.50.5-tier3-fixes-1778291240`

The ABOUT panel's version string was not updated when the build was incremented. Players and QA reading ABOUT will see a stale version number that doesn't match the actual deployed code.

**Fix:** Sync the ABOUT version constant with the build version at deploy time (or derive it from the same source).

---

#### H-02 · HUNT lobby stage art renders blank on second visit

**Reproduction:** Load → navigate to DUEL → navigate to ASCEND → navigate to HUNT. The stage selection panel (stage art + name) is blank. Navigate away from HUNT and back → art renders correctly.

**Scope:** Specific to the pattern of meta-tab navigation before HUNT. First HUNT visit and post-combat return render correctly.

**Impact:** Players who explore tabs before committing to HUNT see a broken lobby.

**Fix required:** HUNT stage panel likely caches a DOM reference on first mount that is invalidated by subsequent tab switches. Force re-render on every tab:change to HUNT, not just on first mount.

### 🟡 MEDIUM — Functional issue with workaround

---

#### M-01 · SETTINGS panel and DAILY reward modal render simultaneously

**Observation:** After closing TASKS modal and clicking 🎁DAILY, both the SETTINGS panel (cfg-close ✕ at screen position x:731, y:28, fully rendered) and the DAILY reward modal were simultaneously visible and interactive.

**Expected:** Opening any modal should either close or lock out other panels.

**Impact:** Players can interact with two overlapping modal surfaces. Z-index layering makes SETTINGS partially obscured but its buttons remain clickable.

**Fix:** Add modal exclusivity logic — any modal/panel open call should close all others first, or use a modal manager with a single-active-modal guarantee.

#### L-01 · Raw ID `mega_bundle` leaking into SHOP → Bundles UI

**Fix:** Set `displayName` for the `mega_bundle` entry in the Bundles catalog.
